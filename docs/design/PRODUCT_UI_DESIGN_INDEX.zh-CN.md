## 目的与使用方式

本目录把既有产品权威展开为可直接实施的全局规则、逐页结构、交互、状态、响应式、组件责任和验收约束。它不重新定义产品，不修改运行时架构，也不把目标能力写成当前能力。

实施人员应按以下顺序使用：先读本索引和全局规范，再读目标页面规范，随后核对组件责任、Naive UI 映射、纠正清单和实施交接。页面规范只记录页面特有决定；共同行为由全局文档唯一拥有，页面以链接引用。

## 权威层级

1. [产品设计蓝图](../product/PRODUCT_DESIGN_BLUEPRINT.zh-CN.md)拥有产品定位、用户、模式、旅程、信息架构、需求、页面责任和验收概念。
2. [所有者需求基线](../product/OWNER_PRODUCT_REQUIREMENT_BASELINE.zh-CN.md)记录稳定的所有者来源；`OD-01` 至 `OD-11` 全部保持 `OPEN`。
3. [产品定义](../product/PRODUCT_DEFINITION.md)、[技术工作区模式](../product/WORKSPACE_MODES.md)、[需求追踪](../product/PRODUCT_REQUIREMENT_TRACEABILITY.json)、[模式能力矩阵](../product/PRODUCT_MODE_AND_CAPABILITY_MATRIX.json)与[产品信息架构](../product/PRODUCT_INFORMATION_ARCHITECTURE.json)分别拥有其声明范围。
4. 当前 `docs/ui/**`、`docs/architecture/**` 与当前运行时代码拥有 Token、组件、状态、数据、持久化和安全的实现事实。
5. 本目录拥有页面级显示、区域内容、交互、状态、响应式、组件组合和实施验收决定；不得覆盖以上权威。
6. `.ai/reports/**`、旧计划和源项目只作证据，不是本目录的活动设计权威。

发现冲突时，停止扩写冲突结论，将其登记到[实施纠正清单](./PRODUCT_IMPLEMENTATION_CORRECTION_BACKLOG.zh-CN.md)，并按上述层级处理。不得以视觉便利越过合同、安全或持久化边界。

## 能力分类

每项页面能力、动作、设置和状态必须且只能使用以下一种分类：

| 分类                  | 含义                                                 | 页面呈现规则                                                   |
| --------------------- | ---------------------------------------------------- | -------------------------------------------------------------- |
| `CURRENT_IMPLEMENTED` | 当前仓库已存在并有运行时所有者                       | 可作为当前能力描述；仍须通过页面验收                           |
| `APPROVED_TARGET`     | 产品已确认，但实现或版本化所有者尚未闭合             | 作为实施目标；不得写成当前可用                                 |
| `CONTRACT_BLOCKED`    | 产品位置明确，但真实 API、认证、实时或权限合同未闭合 | 仅显示真实不可用、登录或权限状态，不发起伪请求，不制造成功内容 |
| `OPEN_OWNER_DECISION` | 有约束或建议起点，但所有者尚未确认                   | 明示 OD 编号、已固定边界与仍为临时建议的内容                   |
| `PROHIBITED`          | 违反产品、架构、安全或来源边界                       | 不显示、不实现，不以隐藏开关或技术入口保留                     |

产品需求追踪中的“确认核心需求”等分类用于需求来源，不等同于运行时实现状态。`unknown` 仅是技术解析的失败关闭哨兵，不是用户可选模式。

## 稳定术语

| 术语       | 定义                                                         |
| ---------- | ------------------------------------------------------------ |
| 统一工作区 | 唯一的左侧来源导航、中央主棋盘、右侧上下文组合               |
| 来源       | 提供 PGN、局面、赛事选择、回放或实时快照的入口               |
| 教学集合   | 一次备课或讲课中组织的一组本地或已导入棋局                   |
| 交接上下文 | 将非敏感来源选择安全传入统一工作区的类型化记录               |
| 本地副本   | 用户显式导入后，与远端只读来源分离的可编辑内容               |
| 实时       | 已连接且数据位于合同确认的新鲜度范围内                       |
| 陈旧       | 保留最后可信数据，但已超过合同确认的新鲜度范围               |
| 合同阻断   | 产品位置存在，但真实合同不足以宣称可用                       |
| 场外大屏   | 独立只读、远距离、可读性优先的赛事展示，不是放大的教学工作区 |
| 教学笔记   | 跟随本地教学棋局的棋局级内容；课次级实体仍由 `OD-02` 决定    |

普通用户文案统一使用“开赛了”“棋谱”“棋局信息”“实时状态”“暂不可用”等产品语言，不出现 DTO、Axios、MQTT、Worker 请求标识、位置哈希或内部错误码。

## 开放所有者决策

`OD-01` 至 `OD-11` 全部保持 `OPEN`。页面可记录建议起点，但必须同时说明已固定边界、临时部分、所有者和复核触发条件。尤其不得静默固定大屏棋盘尺寸、边距、间距、轮播时间、陈旧阈值、棋钟插值、匿名实时范围、窄屏完整编辑、导出范围、AI 资源值或声音默认状态。

## 最终门禁

以下条件同时满足时，执行报告才可给出 `PRODUCT_PAGE_DESIGN_DOCUMENTATION_READY_FOR_IMPLEMENTATION`：

- 全部 19 份 Markdown 文档和七个页面族存在、互相链接且无重复所有权；
- 31 个设计表面、九条规范路由、四个活动技术工作区模式和所有要求状态均有唯一归属；
- 每页具备文本布局、信息层级、动作优先级、滚动、焦点、键盘、触控、减弱动效、响应式、持久化、安全和验收定义；
- 当前、目标、合同阻断、开放决定和禁止能力没有混写；
- 进行中实时无 AI、评价、编辑或来源写回；场外大屏无教学工作区、棋谱分析或编辑控件；
- 所有相对链接、标题锚点和 Mermaid 结构通过检查；全部治理、格式、静态、类型、构建和审计命令通过；
- 变更范围仅为 `docs/design/**` 与指定执行报告，未执行测试、浏览器验证、暂存、提交或推送。

## 后续更新规则

1. 先更新拥有该决定的上游产品、UI 或架构权威，再更新本目录；不得只在页面规范内改变产品能力。
2. 每项新增能力先分类；无真实合同的远端能力只能登记为 `CONTRACT_BLOCKED`。
3. 关闭 OD 必须包含所有者证据、受影响页面、Token/合同/持久化复核和验收更新；建议值不会因实现方便自动变为常量。
4. 新概念组件先进入组件责任规范；只有真实文件合入后才能转为当前实现所有者。
5. 公共规则只修改其全局所有者，页面通过链接继承；页面特有例外必须说明原因、范围和复核条件。
6. 每次更新均运行链接、锚点、Mermaid、分类、路由、OD、范围和仓库治理检查，并同步执行报告。
